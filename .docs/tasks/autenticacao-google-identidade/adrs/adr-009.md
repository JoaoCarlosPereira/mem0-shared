# ADR-009: UI publicada em HTTPS (Traefik) e coexistência redirect + device flow

## Status

Aceito

## Data

2026-07-03

## Contexto

A TI aprovou o DNS `memorias.sysmo.com.br` para a UI, viabilizando o fluxo de
redirect original (ADR-002), que estava bloqueado pelas regras do console do
Google (sem IP puro/HTTP). O device flow (ADR-007) já está funcional e deve
permanecer como alternativa (resiliência a problemas de certificado/DNS e
utilizável por quem acessa pelo IP).

## Decisão

1. **TLS no Traefik existente**: novo entrypoint `websecure` (:443) com
   certificado montado de `${TLS_CERT_DIR:-openmemory/certs}` (`ui.crt`/
   `ui.key`, config dinâmica em `compose/traefik/tls.yml`). Roteamento por
   hostname: `Host(${UI_HOSTNAME}) → openmemory-ui:3000`. O acesso por
   `http://IP:3000` e as rotas MCP/API em `:8765` permanecem intactos.
   Emissão do certificado: Let's Encrypt DNS-01 ou CA interna (runbook em
   `openmemory/certs/README.md`); sem certificado, o Traefik serve o default e
   nada mais é afetado.
2. **Dois fluxos de login na tela**: botão primário "Entrar com Google"
   (redirect, NextAuth — requer a URL https) e alternativa "Entrar com código"
   (device flow). Erro de configuração do redirect orienta o usuário ao código.
3. **Dois clients OAuth no Google** (cada fluxo exige um tipo):
   - `GOOGLE_CLIENT_ID/SECRET` — tipo **Aplicativo da Web**, redirect
     `https://memorias.sysmo.com.br/api/auth/callback/google` (NextAuth).
   - `GOOGLE_DEVICE_CLIENT_ID/SECRET` — tipo **TVs e dispositivos de entrada
     limitada** (device flow); opcional, com fallback para o par principal.
   A validação de ID token aceita `aud` de qualquer client configurado; a
   restrição de domínio (`hd` == `AUTH_ALLOWED_DOMAIN`) permanece única para os
   dois fluxos.

## Alternativas Consideradas

### Alternativa 1: remover o device flow e voltar só ao redirect

- **Por que rejeitada**: o device flow custa zero para manter e cobre falhas de
  certificado/DNS e acesso via IP — resiliência grátis.

### Alternativa 2: um único client OAuth para os dois fluxos

- **Por que rejeitada**: o Google amarra cada fluxo a um tipo de client — o
  endpoint de device code recusa clients Web e vice-versa.

## Consequências

### Positivas

- UX de login com redirect (um clique) para a equipe via https://memorias.sysmo.com.br.
- Fallback por código preservado; agentes MCP intocados.

### Negativas

- Gestão de certificado (renovação ~90 dias no Let's Encrypt) passa a existir.
- Dois pares de credenciais Google para administrar.
