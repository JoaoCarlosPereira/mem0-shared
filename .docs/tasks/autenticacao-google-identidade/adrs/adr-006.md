# ADR-006: Evoluir o TeamAuthMiddleware em middleware unificado de autenticação

## Status

Aceito

## Data

2026-07-02

## Contexto

O `TeamAuthMiddleware` (`app/middleware/team_auth.py`) já intercepta todas as rotas (incluindo `/mcp/*`), extrai credencial de `X-API-Key`/`Authorization: Bearer`, valida contra o mapa de tokens de equipe e popula `team_var` para auditoria, com modos `AUTH_MODE=off|warn|enforce` e métricas `AUTH_OK_TOTAL`/`AUTH_DENIED_TOTAL`. A feature precisa validar três tipos de credencial: token de agente (query `?token=` nas rotas MCP), JWT de sessão da UI (Bearer nas rotas REST) e os tokens de equipe existentes.

## Decisão

Evoluir o `TeamAuthMiddleware` para **`AuthMiddleware` unificado**, mantendo compatibilidade total:

1. **Extração em precedência**: `?token=` (rotas `/mcp/*`) → `X-API-Key` → `Authorization: Bearer`.
2. **Resolução por tipo**: JWT de sessão (assinatura local) → usuário pessoa; token opaco → lookup por hash em `agent_tokens` (com cache Redis, TTL curto) → usuário + máquina + agente; senão → mapa de tokens de equipe (comportamento atual).
3. **Contextvars**: além de `team_var`, novos `auth_user_var`, `machine_var` (hostname da URL MCP validado contra o vínculo) e `auth_method_var` (`agent_token` | `session` | `team` | `legacy`).
4. **Convivência legada**: requisição MCP sem token continua passando (identidade = hostname, `auth_method=legacy`), respeitando `AUTH_MODE` atual; o enforce por grupo (Fase 3) usa o mesmo ponto.

## Alternativas Consideradas

### Alternativa 1: Novo middleware separado para tokens de agente

- **Descrição**: `AgentTokenMiddleware` dedicado a `/mcp`, `TeamAuthMiddleware` intocado.
- **Prós**: Isolamento; risco zero para o comportamento atual.
- **Contras**: Dois pontos de autenticação para manter coerentes (skip lists, modos, métricas); ordem entre middlewares vira nova fonte de bugs.
- **Por que rejeitada**: Duplicação de responsabilidade idêntica.

### Alternativa 2: Validação inline na rota MCP

- **Descrição**: Handler MCP lê e valida `?token=` diretamente.
- **Prós**: Simples e local.
- **Contras**: Não cobre rotas REST; espalha lógica de auth; sem reuso de métricas/modos.
- **Por que rejeitada**: A autenticação precisa ser transversal (MCP + REST).

## Consequências

### Positivas

- Um único ponto de autenticação, com modos e métricas já existentes reaproveitados.
- Identidade resolvida disponível via contextvars para auditoria, métricas de tokens e logging estruturado sem tocar assinaturas de função.

### Negativas

- O middleware ganha dependência de banco/Redis (lookup de token) — mitigada por cache e por lookup lazy (só quando há `?token=`).

### Riscos

- Regressão no comportamento de equipe existente → suíte `test_team_auth.py` mantida e ampliada; modos `off|warn|enforce` preservados byte a byte para credenciais de equipe.
- Latência no hot path MCP → cache Redis do hash validado; meta: overhead ≤ 5 ms p95.

## Notas de Implementação

O mascaramento do `?token=` em logs (ADR-003) é implementado neste middleware antes de qualquer log estruturado, e replicado na config de access log do Traefik.
