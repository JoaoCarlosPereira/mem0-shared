# ADR-003: Token de agente transportado na URL MCP (query param)

## Status

Aceito

## Data

2026-07-02

## Contexto

Os agentes MCP (claude-code, cursor, codex) são configurados pela receita do `GET /provision`, que hoje gera apenas `type`/`url` — a identidade vai posicional na URL (`/mcp/{client_name}/http/{hostname}`), sem headers. O suporte a headers customizados varia por cliente (o config TOML do codex, por exemplo, não tem suporte garantido). O PRD exige que o agente se autentique com um token por usuário.

## Decisão

O token de agente viaja **exclusivamente como query param na URL MCP** gerada pelo provision:

```
{base_url}/mcp/{client_name}/http/{hostname}?token=<token>[&group=...]
```

Compensações obrigatórias por o token estar em URL:

- O backend **nunca** loga a URL crua de rotas `/mcp`: mascaramento do parâmetro `token` em logs de acesso, logs estruturados e traces (`token=***`).
- O token não entra em `read_audit_logs`/`write_audit_logs` (apenas o prefixo identificador, quando necessário).
- Revogação e regeneração de baixo atrito pela UI compensam o risco maior de exposição em arquivos de config.

## Alternativas Consideradas

### Alternativa 1: Header `Authorization: Bearer` + fallback query

- **Descrição**: Provision emite campo `headers` para clientes que suportam; query param só como fallback.
- **Prós**: Token fora da URL para a maioria dos clientes; padrão de mercado.
- **Contras**: Dois canais para manter; suporte a headers varia por cliente e versão; receita do provision mais complexa por host.
- **Por que rejeitada**: O usuário priorizou simplicidade e compatibilidade universal — um único canal que funciona em todos os clientes.

### Alternativa 2: Somente header

- **Descrição**: Token exclusivamente via `Authorization`/`X-API-Key`.
- **Prós**: Mais seguro (nada em URL).
- **Contras**: Clientes MCP sem suporte a headers ficam de fora.
- **Por que rejeitada**: Quebraria a compatibilidade com parte dos clientes usados pela equipe.

## Consequências

### Positivas

- Funciona em qualquer cliente MCP sem depender de suporte a headers.
- Receita do provision permanece simples (uma URL por host).

### Negativas

- Token fica gravado nos arquivos de config dos agentes (`.mcp.json`, `config.toml`) e pode aparecer em URLs de proxies intermediários (Traefik) — exige mascaramento em toda a cadeia de logs.

### Riscos

- Vazamento do token via log não mascarado → checklist de mascaramento (API, Traefik access log, OTel) como critério de aceite; rotação imediata disponível na UI.

## Notas de Implementação

A validação ocorre no middleware unificado (ADR-006), que extrai `?token=` das rotas `/mcp/*` antes de qualquer log de negócio.
