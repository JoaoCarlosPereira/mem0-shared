# ADR-004: Modelo de dados — novas tabelas `machines` e `agent_tokens`, colunas de identidade em `users`

## Status

Aceito

## Data

2026-07-02

## Contexto

Hoje a tabela `users` mistura conceitos: cada linha representa uma máquina (campo `user_id` guarda o hostname). O PRD separa três conceitos — Usuário (pessoa), Máquina (computador) e Agente (processo local) — e exige vínculo auditável entre eles, preservando a base legada e evitando duplicação.

## Decisão

Evolução aditiva do esquema (nova revision Alembic com `down_revision=f1a2b3c4d5e6`):

- **`users` ganha colunas**: `google_sub` (String, único, nullable), `display_name`, `avatar_url`, `user_type` (`person` | `legacy_host`, default `legacy_host` no backfill). Linhas legadas (hostname) permanecem intactas como `legacy_host`.
- **Nova tabela `machines`**: `id` (UUID PK), `hostname` (String único, indexado), `linked_user_id` (FK→users.id, nullable), `legacy_user_id` (FK→users.id, nullable — a linha legada correspondente), `status` (`unlinked` | `linked` | `conflict`), `linked_at`, `linked_by` (FK→users.id), `created_at`, `updated_at`.
- **Nova tabela `agent_tokens`**: `id` (UUID PK), `user_id` (FK→users.id, único enquanto ativo — um token por usuário), `token_hash` (String, indexado), `prefix` (String curto não-secreto, ex.: `omtk_2F9K`), `created_at`, `revoked_at` (nullable), `last_used_at` (nullable, Fase 2).
- **Nova tabela `link_audit_logs`**: registro de vínculos/desvínculos/conflitos (`machine_id`, `actor_user_id`, `action`, `detail`, `created_at`).

## Alternativas Consideradas

### Alternativa 1: Reusar `users` + tabela única de vínculo

- **Descrição**: FK `legacy_user_id → person_user_id` na própria `users` + `agent_tokens`.
- **Prós**: Menos tabelas.
- **Contras**: Mistura pessoa e máquina na mesma entidade; complica múltiplas máquinas por pessoa (Fase 2) e o estado de conflito.
- **Por que rejeitada**: Ambiguidade conceitual — exatamente o problema que a feature quer eliminar.

### Alternativa 2: Esquema de identidade completo (identities, credentials, memberships)

- **Descrição**: Módulo de identidade genérico e extensível.
- **Prós**: Flexível para futuros provedores/papéis.
- **Contras**: Sobre-engenharia para uma equipe interna com um provedor e sem papéis granulares.
- **Por que rejeitada**: YAGNI.

## Consequências

### Positivas

- Os três conceitos do PRD têm representação explícita e auditável.
- Migração 100% aditiva — nenhum dado legado alterado ou movido.
- Estado de conflito modelado em `machines.status`, pronto para a tela da Fase 2.

### Negativas

- `users` passa a conter dois tipos de linha (`person` e `legacy_host`) até a migração completar — consultas existentes precisam ignorar a distinção (comportamento atual preservado pelo default).

### Riscos

- Consultas legadas que assumem `users.user_id = hostname` → manter o campo intocado; `person` usa `user_id = google_sub` como chave técnica, sem colidir com hostnames.

## Notas de Implementação

Token em claro nunca é persistido: apenas `token_hash` (SHA-256) e `prefix`. A unicidade "um token ativo por usuário" é garantida por índice parcial (`WHERE revoked_at IS NULL`).
