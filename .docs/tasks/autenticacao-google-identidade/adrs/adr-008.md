# ADR-008: Token de agente imutável e permanentemente exibível

## Status

Aceito

## Data

2026-07-03

## Contexto

O desenho original (ADR-003/ADR-004) tratava o token de agente como segredo
rotacionável: apenas o hash persistido, valor em claro exibido uma única vez,
revogação/regeneração pela UI. Na revisão de UX o usuário decidiu o oposto para
este deploy interno: o token deve ser **gerado automaticamente no primeiro
login, atribuído imutavelmente à conta e exibido permanentemente** na tela de
comandos de instalação — prioridade total à simplicidade operacional (instalar
agentes em várias máquinas a qualquer momento, sem cerimônia de "copie agora").

## Decisão

1. **Get-or-create idempotente**: `POST /api/v1/agent-token` cria o token na
   primeira chamada e, nas seguintes, devolve **o mesmo** valor — nunca
   rotaciona. `GET` também devolve o valor em claro (não só metadados).
2. **Imutável via API**: o endpoint `DELETE` (revogação) foi removido; não há
   regeneração pela UI. `revoked_at` permanece no modelo como válvula de
   emergência **administrativa** (UPDATE direto no banco) — o middleware
   continua recusando tokens revogados.
3. **Valor recuperável**: nova coluna `agent_tokens.token_value` guarda o claro
   (migração `h3c4d5e6f7a8`); o `token_hash` continua sendo o índice de
   autenticação do middleware (nada muda no hot path).
4. **UI**: a tela de instalação (dashboard) gera o token automaticamente na
   primeira visita e o exibe embutido nos comandos **para sempre**; o painel
   `/settings/install` vira consulta/cópia (sem revogar/regenerar).

## Alternativas Consideradas

### Alternativa 1: manter show-once + rotação (desenho original)

- **Prós**: segredo nunca recuperável do banco; rotação de baixo atrito.
- **Contras**: usuário que perde o valor precisa rotacionar e reconfigurar TODAS
  as máquinas; atrito recorrente para instalar novos agentes.
- **Por que rejeitada**: decisão explícita do usuário — em LAN interna, a
  conveniência de instalação vale mais que a propriedade "hash-only".

### Alternativa 2: exibir sempre, mas manter rotação opcional na UI

- **Por que rejeitada**: "imutável" foi requisito explícito; rotação na UI
  contradiz a expectativa de que os comandos exibidos continuem válidos para
  sempre.

## Consequências

### Positivas

- Instalação de agentes em qualquer máquina, a qualquer momento, sem depender
  de ter guardado o valor.
- Fluxo de primeiro login sem passos manuais (token nasce sozinho).

### Negativas (trade-off aceito conscientemente)

- O valor em claro fica no banco (PostgreSQL interno): quem tem acesso de
  leitura ao banco lê os tokens. Aceito para o contexto LAN (o banco já guarda
  todo o conteúdo das memórias; o perímetro é o mesmo).
- Sem rotação self-service: em suspeita de vazamento, a resposta é
  administrativa (`UPDATE agent_tokens SET revoked_at = now() WHERE ...` e
  gerar novo via nova linha) — documentado como runbook.

### Riscos

- Token vazado permanece válido até intervenção manual → mitigação: mascaramento
  de logs (ADR-003) permanece integral; trilha de auditoria identifica uso.
