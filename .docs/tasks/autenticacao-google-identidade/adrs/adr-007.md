# ADR-007: Login Google via Device Flow (sem URL de redirect)

## Status

Aceito

## Data

2026-07-03

## Contexto

O ADR-002 definiu o login Google via fluxo de autorização com redirect (NextAuth).
No deploy real descobriu-se um bloqueio: a UI roda em `http://192.168.3.213:3000`
(rede interna), e o console do Google **rejeita** URLs de redirect com IP puro e
com `http://` (exceções só para localhost). As saídas com redirect exigiriam DNS
interno + certificado TLS — dependência da TI ainda não aprovada. Além disso, a
criação de projetos no Google Cloud está restrita ao admin do Workspace.

## Decisão

Adotar o **OAuth 2.0 Device Flow** ("TVs e dispositivos de entrada limitada")
como mecanismo de login, complementando o ADR-002 (que permanece válido para o
futuro, se DNS/TLS forem aprovados):

1. `POST /api/v1/auth/google/device/start` — a API pede um código ao Google
   (`oauth2.googleapis.com/device/code`) e devolve `user_code` +
   `verification_url` + `device_code` à UI.
2. A UI exibe: "Acesse google.com/device e digite o código XXXX-XXXX" e faz
   polling em `POST /api/v1/auth/google/device/poll`.
3. O usuário autoriza em qualquer navegador/celular; a API recebe do Google o
   **mesmo ID token** do fluxo original, validado pelo mesmo caminho
   (assinatura, `aud`, `exp`, claim `hd` = domínio) e emite o mesmo JWT de sessão.
4. A sessão NextAuth é estabelecida por um provider **Credentials** ("device"):
   o `authorize()` valida o JWT chamando `GET /auth/me` na API — a fonte da
   verdade continua sendo o backend.

A credencial no Google Cloud passa a ser do tipo **"TVs e dispositivos de
entrada limitada"** (sem campo de redirect). O `GOOGLE_CLIENT_SECRET` passa a
ser necessário também na API (o polling do token exige id+secret).

## Alternativas Consideradas

### Alternativa 1: DNS interno + TLS (manter só o redirect do ADR-002)

- **Descrição**: `memorias.sysmo.com.br` → IP interno + certificado (DNS-01).
- **Prós**: UX de redirect automática; padrão web.
- **Contras**: Depende de aprovação/execução da TI (DNS + certificado); bloqueia o rollout.
- **Por que rejeitada (agora)**: dependência externa ao time; o Device Flow destrava sem ela. Pode ser adicionada depois — os fluxos coexistem.

### Alternativa 2: IdP local (Keycloak)

- **Descrição**: servidor de identidade na própria stack; redirect para IP interno permitido.
- **Prós**: zero dependência de internet.
- **Contras**: mais um serviço para operar, provisionamento de usuários, abandona a identidade Google decidida no PRD.
- **Por que rejeitada**: o usuário confirmou que egress de autenticação é aceitável; a premissa é só sobre o conteúdo das memórias.

### Alternativa 3: hosts/sslip.io + certificado autoassinado

- **Por que rejeitada**: configuração manual por PC e avisos de certificado — custo de suporte contínuo.

## Consequências

### Positivas

- Login Google funcional em IP interno com HTTP, sem DNS, sem TLS, sem TI.
- Reuso integral da validação de ID token, onboarding, tokens de agente e middleware.
- Coexistência futura com o fluxo de redirect (upgrade de conveniência).

### Negativas

- UX com um passo extra (digitar o código em google.com/device) — mitigado pela sessão de 7 dias.
- `GOOGLE_CLIENT_SECRET` presente também no ambiente da API (antes só na UI).

### Riscos

- `device_code` trafega para o browser na LAN (design stateless p/ múltiplos workers): vida curta (~30 min), uso único e mesmo canal do JWT de sessão — risco equivalente ao já aceito.
- Polling excessivo → respeitar `interval` e o erro `slow_down` do Google.

## Notas de Implementação

Requisições ao Google isoladas em função monkeypatchável (`_post_form`) — testes sem rede. O endpoint de start é fail-closed (503 sem client id/secret/domínio).
